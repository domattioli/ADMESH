# Research: Octree O(log N) Scalability

**Feature**: specs/021-octree-size-field-perf | **Date**: 2026-06-02

## Decision 1: Tree traversal strategy for point location

**Decision**: Recursive tree descent using bbox midpoint split rule (`pt[d] < mid[d]` → left child).  
**Rationale**: Natural O(depth) = O(log N) cost. Midpoint split is already used by the prototype's `split()` rule — descent is simply its inverse. No additional data structure needed beyond child pointer array.  
**Alternatives considered**:
- KD-tree (`scipy.spatial.cKDTree`): O(log N) queries but adds external dep, breaks prototype's internal structure
- R-tree: Overkill for regular quadtree; fixed split rule means midpoint descent is sufficient
- Linear scan (status quo): O(N) — confirmed intractable at ratio 1000

## Decision 2: Adjacency strategy for 2:1 balancing

**Decision**: Build neighbour links during `split()` via parent/sibling relationships. No post-hoc all-pairs pass.  
**Rationale**: Each `split()` produces exactly 4 children. Sibling links among the 4 are trivially O(1) to set. Cross-parent neighbours can be located in O(depth) by ascending parent links. Total adjacency cost across all N splits: O(N log N) worst case, effectively O(N) in practice (tree is shallow).  
**Alternatives considered**:
- Rebuild full adjacency matrix after each split (status quo): O(N²) — the root cause
- Spatial hash: O(1) expected but hash collisions and bucket management add implementation complexity without meaningful advantage over pointer links for a regular quadtree

## Decision 3: Work-queue balancing vs. pass-based

**Decision**: Single-pass work-queue (`collections.deque`). Split a violating leaf, enqueue new violation candidates (O(4) per split). Repeat until queue empty.  
**Rationale**: O(N) total work. Eliminates the O(N³) worst case from the prototype's "rebuild adjacency per split iteration" pattern. Standard algorithm for 2:1 balanced quadtrees (Samet 1990 §2.3).  
**Alternatives considered**:
- Multi-pass scan (status quo): O(N²) per pass, O(N³) total — root cause of build intractability
- Morton code ordering + bitwise neighbour lookup: O(1) neighbour lookup but requires coordinate encoding and loses pointer-link generality needed by the distmesh integration

## Decision 4: Numba vectorization scope

**Decision**: P1 optional, gated behind `ADMESH_NUMBA=1` env var. Scope: `octree_medial.py` per-leaf gradient fit loops only.  
**Rationale**: Phases P0–P3 deliver the primary O(log N) scaling. Numba adds first-run latency (1–3 s) that is unacceptable for interactive use. The `ADMESH_NUMBA=1` pattern already exists in `_solve_iter_nb` — reuse is cleaner than a new gate mechanism.  
**Alternatives considered**:
- Always-on Numba: Rejected (latency in CI + interactive sessions)
- CuPy/GPU: Out of scope (issue #8)
- Cython: Higher maintenance cost than Numba for equivalent speedup

## Decision 5: Parity test reference data

**Decision**: Reference output from `tests/fixtures/octree/river-into-bay*.npz` generated by the prototype. Tolerance `atol=1e-10`.  
**Rationale**: The prototype (`_build_adjacency` + linear scan) is known-correct algorithmically. The pointer-tree rewrite must reproduce its numerical output identically (up to float tie-breaking). `atol=1e-10` is the ADMESH numerical-identity standard (same as faithful-port parity tests).  
**Alternatives considered**:
- Exact equality (`atol=0`): Too strict; float tie-breaking at cell boundaries may differ by < machine epsilon
- `atol=1e-6`: Too loose; could mask regressions in the gradient-limited size field
