# Implementation Plan: ADCIRC Mesh Registry

**Branch**: `005-adcirc-mesh-registry` | **Date**: 2026-04-25 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/005-adcirc-mesh-registry/spec.md`

## Summary

Build a federated registry of ADCIRC-compatible coastal-simulation meshes,
delivered as **three loosely-coupled deliverables**:

1. A **GitHub repo** (`domattioli/adcirc-mesh-registry`, separate from
   ADMESH) holding a **TOML manifest** as the single source of truth, with
   PR-based contribution and CI validation.
2. A **HuggingFace Datasets mirror** publishing the manifest plus mesh
   files for redistributable licenses, generated by a release-tag
   GitHub Action.
3. A **Python loader package** (`adcirc-mesh-registry` on PyPI) exposing
   `registry.find(...)` and `Mesh.load()` for downstream tools.

Phase 1 of this spec ships all three deliverables for a bootstrap of ~5
seed meshes; subsequent phases (HF pipeline polish, AI-assisted bundles)
are out of scope here and tracked in the Phasing section of the spec.

## Technical Context

**Language/Version**: Python 3.10+ (matches ADMESH; allows `tomllib` from
the stdlib without a backport dependency).
**Primary Dependencies**: `tomllib` (stdlib, read), `tomli-w` (write,
~6 KB pure Python), `pydantic` v2 (schema validation, JSON-schema
emission for HF dataset card), `shapely` v2 (bbox overlap), `httpx`
(file fetching for the loader, async-capable, no system OpenSSL
gotchas), `huggingface_hub` (publish to HF Datasets), `click` (CLI for
local validation: `mesh-registry validate manifest.toml`).
**Storage**: TOML files in the GitHub repo (single `manifest.toml`
until 5K entries, then sharded as `manifests/<namespace>.toml`). Mesh
files mirrored to HuggingFace Datasets for redistributable licenses;
proprietary/unknown remain at original `source_url`.
**Testing**: `pytest` with `pytest-asyncio` (httpx async paths) and
`responses`/`respx` for HTTP mocking. Schema-roundtrip tests via
hypothesis (TOML write → read → equality).
**Target Platform**: Pure-Python wheel, `pip install`-able on
Linux/macOS/Windows, Python 3.10–3.13. CI runs on
`ubuntu-latest`/`macos-latest`/`windows-latest`.
**Project Type**: Library + CLI + GitHub Actions templates. NOT a
service — there is no server-side component beyond GitHub itself and
HuggingFace Datasets (both managed).
**Performance Goals**: `registry.find(...)` returns in under 1 second
for the full 10K-entry catalog when run against a locally cached
manifest (FR-014, SC-009). Manifest parse cold-start budget: 200 ms
for 10K entries on a modern laptop CPU.
**Constraints**: No C extensions (consistent with ADMESH constitution
Principle II, even though this isn't a port). No mandatory
authentication for the loader (HF public-dataset access is anonymous).
TOML schema is forward-compatible: new optional fields don't break
older readers (FR-001 + Assumptions).
**Scale/Scope**: ~10K mesh entries at maturity; ~5 at Phase 1 launch
(SC-002). Per-mesh files 0.1–100 MB; HF mirror estimated 1–100 GB at
maturity (mostly under HF's free-tier dataset limits).

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

The constitution (`.specify/memory/constitution.md`) declares Principle I
(faithful port) as the **default null hypothesis** and requires non-port
features to be explicitly justified. This feature is non-port; the
following table addresses each principle.

| Principle | Status | Notes |
|---|---|---|
| I. Faithful Port Before Optimization | **N/A — justified** | This is a community data-engineering tool, not a translation of a MATLAB module. There is no MATLAB source under `01_ADMESH_Library/` to port; the registry is an ecosystem deliverable that complements ADMESH but does not replace any existing functionality. The 13 stage modules and the `001-pythonize-and-fort14-integration` additions remain untouched. |
| II. Pure-Python First | **PASS** | All chosen dependencies are pure Python or have pure-Python wheels on every supported platform. No C extensions in the loader. (`shapely` v2 ships manylinux/win/macos wheels and is permitted by the same logic ADMESH uses for `scipy`.) |
| III. Reference-Test Discipline | **PASS (adapted)** | This feature has no MATLAB reference. Equivalent discipline: golden-file fixtures (small TOML manifests + expected query results) under `tests/fixtures/registry/`. CI gates the full suite. |
| IV. Stage-by-Stage Bottom-Up | **PASS (adapted)** | The three deliverables are sequenced bottom-up: schema/validator first, loader second, HF publisher last. Each ships with its own tests before the next is started. |
| V. Report-and-Advance Cadence | **PASS** | Operational; applies to session work, not to artifact design. |

**Gate result**: PASS with documented N/A on Principle I (see Complexity
Tracking below for the explicit deviation rationale).

### Source Code (target repo: `domattioli/adcirc-mesh-registry`)

The implementation lives in a **new repository**, not inside ADMESH.
Per the spec ("federated, not owned by any single project"), the
registry is a peer to ADMESH, not a sub-feature of it. ADMESH may
later depend on the published `adcirc-mesh-registry` package via
`pyproject.toml`, but no code from this feature lands in `admesh/`.

```text
adcirc-mesh-registry/                 # new repo
├── manifest.toml                     # source of truth (single file <5K entries)
├── manifests/                        # sharded layout (≥5K entries; FR-013)
│   └── <namespace>.toml
├── meshes/                           # optional local stash for self-hosted seeds
│   └── <namespace>/<name>/<version>.fort.14
├── src/adcirc_mesh_registry/         # importable Python package
│   ├── __init__.py                   # re-exports: find, Mesh, load_manifest
│   ├── schema.py                     # pydantic models: Mesh, MeshOperation, License
│   ├── manifest.py                   # load/write TOML, sharded resolution
│   ├── query.py                      # find(...) implementation, bbox overlap
│   ├── loader.py                     # Mesh.load() — fetch+cache from HF or source_url
│   ├── publisher.py                  # HuggingFace Datasets publishing
│   └── cli.py                        # `mesh-registry validate|publish|find`
├── tests/
│   ├── fixtures/                     # golden TOML manifests + expected results
│   ├── test_schema.py
│   ├── test_manifest.py
│   ├── test_query.py
│   ├── test_loader.py
│   └── test_publisher.py
├── .github/
│   └── workflows/
│       ├── validate-pr.yml           # CI: schema + sanity checks on PR
│       └── publish-hf.yml            # release-tag: push to HuggingFace Datasets
├── pyproject.toml
├── README.md
└── CONTRIBUTING.md                   # PR-based submission walkthrough
```

**Structure Decision**: New repo, single Python package (no monorepo
split). The CI workflows and the loader package coexist because they
share schema definitions (`schema.py`); duplicating them across two
repos would invite drift. The repo is structured so that the package
is `pip install`-able independently of the manifest data — the
manifest is data the package operates on, not data the package
embeds.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| New repo (vs. ADMESH submodule) | The spec explicitly calls for "federated, not owned by any single project." If the registry lives inside ADMESH, ADMESH becomes a de-facto governance body for a community catalog — wrong incentive structure, and creates a circular dependency the moment ADMESH wants to consume the registry. | A submodule under `admesh/registry/` would couple registry releases to ADMESH releases and force registry contributors to fork ADMESH. |
| Principle I deviation (non-port feature) | Justified above: there is no MATLAB source for a community catalog. The registry is additive ecosystem infrastructure. | Treating this as a port would require fabricating a MATLAB precedent that doesn't exist, defeating the purpose of the principle. |
